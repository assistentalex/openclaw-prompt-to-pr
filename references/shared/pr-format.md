# PR Format

Conventions for branches, commits, and Pull Request bodies.
Applied in FAZA 6 — PR, after explicit user approval.

---

## Branch Naming

Format: `{type}/{slug}`

| Mode | Type prefix | Example |
|---|---|---|
| 🚀 New Feature | `feat/` | `feat/email-verification` |
| 🐛 Bug Fix | `fix/` | `fix/email-not-sending-on-register` |
| ♻️ Refactor | `refactor/` | `refactor/user-service-cleanup` |
| 🧪 Test Coverage | `test/` | `test/user-service-coverage` |
| 📖 Document | `docs/` | `docs/api-authentication` |
| 🔍 Code Review | N/A | (read-only, no branch) |

Slug rules:
- Lowercase, hyphens only, no underscores
- Max 40 characters
- Derived from task description (not from the user's raw prompt)

```bash
git checkout -b feat/email-verification
```

## Commit Message — Conventional Commits

Format:
```
{type}({scope}): {description}

{body — optional, what and why}

{footer — optional, breaking changes, closes issues}
```

Types:
| Type | When |
|---|---|
| `feat` | New feature |
| `fix` | Bug fix |
| `refactor` | Code cleanup, no behavior change |
| `test` | Adding or fixing tests |
| `docs` | Documentation only |
| `chore` | Build, deps, config changes |
| `perf` | Performance improvement |

Scope: the module or area changed (`auth`, `user`, `payment`, `api`)

Examples:
```
feat(auth): add email verification flow

Sends a verification email after registration.
User must verify before accessing protected routes.

Closes #42
```

```
fix(email): prevent duplicate sends on retry

Email service was not checking idempotency key,
causing duplicate sends on network timeout retry.
```

Rules:
- Description: imperative mood ("add" not "added", "fix" not "fixed")
- Max 72 characters on first line
- Body explains *why*, not *what* (the diff shows what)

## PR Body Template

```markdown
## What

{1–3 sentences describing the change}

## Why

{1–3 sentences explaining the motivation or problem solved}

## Changes

- `src/services/email.service.ts` — added `sendVerificationEmail()`
- `src/services/user.service.ts` — calls email service on register
- `src/routes/auth.routes.ts` — new `/verify-email` endpoint
- `prisma/schema.prisma` — added `emailVerified` field to User

## How to test

1. Register a new user
2. Check email inbox for verification link
3. Click link → should redirect to `/verified`
4. Try accessing protected route before and after verification

## Test results

- Suite: Jest — 47 passed, 0 failed
- New tests: 3 added in `tests/auth/email-verification.test.ts`
- Coverage: 71% → 78%

## Notes

{Any known limitations, follow-up issues, or decisions made}

---
*Generated by prompt-to-pr*
```

## Per-mode PR adjustments

### 🐛 Bug Fix
Add a "Root Cause" section:
```markdown
## Root Cause
{Explanation of what caused the bug and why this fix addresses the root cause, not just the symptom}
```

### ♻️ Refactor
Add a "Behavior Unchanged" section:
```markdown
## Behavior Unchanged
All {N} existing tests pass unchanged.
No public API signatures were modified.
No database schema changes.
```

### 🧪 Test Coverage
Emphasize coverage delta:
```markdown
## Coverage Delta
| Module | Before | After |
|---|---|---|
| user.service | 45% | 82% |
| auth.middleware | 60% | 91% |
| Overall | 62% | 74% |
```

### 📖 Document
List what was documented:
```markdown
## Documented
- 12 functions missing docstrings → added JSDoc/docstrings
- README.md → updated Getting Started section
- API.md → added authentication endpoints
```

## Local runtime note

`tasks/state.json` and `tasks/todo.md` are local runtime working files.
Do not link to them in PR bodies as if they were committed repository artifacts.
If plan context matters in the PR, summarize it directly in the PR body instead.

## Git commands sequence

```bash
# 1. Create branch
git checkout -b {branch-name}

# 2. Stage changes (only files from the plan — never git add .)
git add {file1} {file2} {file3}

# 3. Verify what's staged
git diff --staged --stat

# 4. Commit
git commit -m "{type}({scope}): {description}"

# 5. Push (only after user approves)
git push origin {branch-name}

# 6. Create PR (if gh CLI available)
gh pr create --title "{type}({scope}): {description}" --body "{PR body}"
```

**Never use `git add .`** — only stage files explicitly listed in the plan.
**Never force push** to existing branches without warning the user.
